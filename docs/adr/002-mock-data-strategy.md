# ADR-002: Mock 데이터 전략

## 상태
승인

## 맥락
실제 ENOVIA 시스템의 데이터에 접근할 수 없으므로, K-FCS 화력통제시스템을 시뮬레이션하는 Mock 데이터가 필요하다.

## 결정
방산 도메인을 정확히 반영하는 구조화된 Mock 데이터를 설계한다.
- BOM 트리: 체계(System) → 부체계(Subsystem) → 조립체(Assembly) 3계층
- SW 모듈 11개: 실제 FCS 구성요소 반영 (탄도계산, 사격통제, 안전연동, 센서융합 등)
- ECO 3개: 다양한 상태(draft/review/approved)와 우선순위(critical/high/medium)
- 과거 이력 5개: RAG 기반 지식 검색용 (발생 이슈 + 교훈 포함)

## 근거
- Mock이지만 실제 BOM 구조(EBOM)와 동일한 트리 구조 사용
- 모듈 간 의존성을 정확히 모델링 → 영향 분석의 현실성 확보
- 각 모듈에 실제 C/C++ 코드 스니펫 포함 → 코드 생성 품질 향상

## 프로덕션 대안
- ENOVIA 3DEXPERIENCE Web Services API로 실시간 BOM/ECO 조회
- Git 저장소 연동으로 실제 소스 코드 참조
- 데이터 계층이 분리(data/ 디렉토리)되어 있어 교체 시 컴포넌트 수정 불필요

## 설계자의 실무 경험 반영
- DOM Tree 설계 경험 → BOM 트리 구조 설계에 직접 활용
- IMS 200건+ 이슈 처리 경험 → ECO 상태/우선순위 체계 설계
- 인수인계 문서 작성 경험 → 과거 이력의 "교훈" 필드 설계
